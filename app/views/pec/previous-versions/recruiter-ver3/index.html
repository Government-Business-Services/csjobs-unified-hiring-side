V3

V3 UR (Mural board)

Design questions

Would iterating the 'single view' design into a tabbed layout allow the user easier access to more info / functionality?

There is a lot of information in PEC and a lot of user needs requiring quick and easy access to info and functionality. It was felt that by 'compartmentalising' the info in a way that users could move quickly between info / function types that it would make PEC progress both quicker and easier.

What we tried

We used the tabs component to sectionalise the PEC info into easily accessible chunks. Not only could users navigate around each tab as needed, but some tab content could link to other tabs directly (e.g. the 'Pre-employment checks' tab linked to the 'candidate PEC info' tab when a user asked to view the info entered for a particular check), saving users from bouncing between tabs when looking for info.

What we found

Could we design a way to reuse the result of a previous check, saving time for all users?

If a candidate has more than one application in, or has recently applied for another govt job, then a particular check may have already been done under that other application. If the result of that existing check (known as a 'duplicate check') could be reused, it would save the candidate re-entering the same information, and the checker having to do the same check again.

What we tried

We created a version of the prototype where a duplicate check warning would appear to the checker. This would then give them additional task options - view the duplicate check (to see the result, how long ago it was, which application it was done under), and to ask the vacancy holder of the current application if the check result could be reused rather than done from scratch.

The duplicate check warning and options were shown in a visually obvious ticket panel component which was adapted to show a red eyecatching edge.

Permission could be asked by sending a message, and receiving confirmation back. This would then allow the user to mark the decision on the check and prevent duplicate effort.

What we found

Would highlighting new contact from another user be useful?

Timely and easy communication is seen as a major step towards avoiding the progress 'black hole' where it is difficult for any user to know what is happening with a particular check. We believed that we could help this by not only giving a 'Contact' tab where users could send or receive messages, but also by emphasising when a new message was available. This would allow the reader to see and action the message as soon as possible.

What we tried

We used the HMRC and MOJ Notification Badge component to show that a new message was available on the 'Contact' tab, leading the user to the tab where the same component flagged up a new message that could be read.

What we found

V2

V2 UR (Mural board)

Design questions

Would a simple dashboard (showing a list of PECs, their progress, and the option to give each one a status when complete) be good enough for users?

Firstly, we considered a risky assumption - that a fully fledged system for PECs was not actually needed at all, and instead a simple 'checklist' page which a user could tick when checks were completed would be an improvement on the 'as is' situation.

What we tried

We asked users whether a simple checklist would be be beneficial rather than building something more complex.

What we found

  • this would not offer much improvement (if any) for the main user types - they already had a clunky means of seeing check status in vX
  • it did not solve the problem of users needing multiple systems to action tasks, check progress etc
  • it may have had limited benefit for vacancy holders and candidates if they could access it to see progress of checks

Can a task list view of PECs work for all users: checkers, managers, vacancy holders, candidates?

The IxD viewpoint was that a single design (task list and pages hanging off this) could be used for all user types - e.g. checkers, managers, vacancy holders, candidates - rather than design a different set of screens for each user type. The 'single view' design would allow page elements to be shown, hidden or restricted depending on the user that was logged in (with the assumption that login credentials would drive what could and could not be seen / done).

What we tried

We iteration the V1 task list to create a checker-friendly task list and tested it with checkers and non-checkers, to see if all users could use it successfully (plans were made to test it with candidates in V4).

What we found

  • the 'single view' was easily used and widely liked by all users regardless of their role
  • the simple and clean design was a significant improvement over current systems like vX
  • showing or hiding elements could easily be done to cater for relevant user group
  • showing checks that were not needed was worthwhile for reassurance

Would all workflow allocation types - single task allocated, multiple tasks allocated, entire candidate allocated - be feasible?

As checkers are one of the biggest PEC user groups, we wanted to challenge how they were currently allocated work (one isolated task at a time) with two other workflow options: giving them a choice from multiple tasks, and allocating an entire candidate - and all of the checks that candidate needed - to the checker.

What we tried

We designed three workflow dashboards - a single task, multiple tasks and an entire candidate - and tested all three with users. The designs all used the tabs component to show both available tasks and user work times (to replicate what many are currently used to on the Pre-employment Allocation tool).

What we found

  • users understood all allocation designs
  • there was no significant preference for any of the workflow suggestions - opinions were quite individual and sometimes depended on experience (e.g. had worked a candidate view before) or motivational (e.g. wanted to see a list of tasks going down as it was worked)
  • we should consider whether giving users a choice when signing in would cover all needs

Would access to comms - and the ability to reply to them - be useful?

A significant issue for users like checkers is that they have to use multiple systems to send and receive comms with candidates, vacancy holders etc. We wanted to see if building in some basic comms info would be beneficial.

What we tried

We added fictional emails into the timeline, with the option to reply to that email from the timeline.

What we found

  • users often spotted the email in the timeline
  • they liked the opportunity to reply there and then
  • showing a full email in the timeline could be overkill, especially if there was a long email conversation leading to multiple entries in the timeline

Other work - iterations from V1

  • expanding timeline on task list page was dropped, as the need for subsequent task options meant jumping to another page would usually be necessary, defeating the purpose of quickly showing the timeline
  • timeline content was updated to try and reflect more real-world scenarios for PECs
  • key details bar was improved - reduced in size, improved content e.g. candidate's email address
  • status tag content was improved to show outcome of check e.g. PASSED, RISK ASSESSMENT NEEDED
  • ability to change task status was added

V1

V1 UR (Mural board)

Initial design was agreed with Service Designer (Oceanne) and limited to:

  1. sign in page
  2. 2FA page
  3. user dashboard
  4. list of PECs

Was agreed to reuse existing design of pages 1-3 from existing campaign / sift prototype, as these had been extensively tested already. The designs were not refined for PEC, as they were simply to drive conversation about what a user would want to see on each page.

Page 4 (list of PECs) utilised the existing task list pattern, as (a) it was a standard pattern and (b) it had already been used and tested in campaign / sift prototype.

The DWP Key Details Bar - already tested on agent-facing services - was added to give prominent info on the candidate's details at all times.

The user was given two designs for viewing historical info on each check. Both designs used the same timeline component to display the info, as this was a tested pattern and the IxD (Jonathan) had also worked on DWP services where this had been extensively tested (e.g. Winter Fuel Payments, Get your State Pension). The timeline was populated with dummy content to drive requirements from the user during testing.

The first design used the Reveal pattern to show the timeline when the task name was clicked (the reveal was amended to hide the arrow icon to retain the task list look and feel).

The second design took the user to a new page which displayed the same timeline info.

At all relevant points, the tag component was used to indicate status of the check at that point.
